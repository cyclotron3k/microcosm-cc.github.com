Standard Behaviour

In general the API supports a basic set of functionality that allows you to control the way in which the API will respond to your requests.

Boiler Plate

Our responses are wrapped in boiler plate JSON that helps to describe what has happened.

{
  context: "",
  status: 200,
  data: null
  error: null
}

This boiler plate can be disabled by ensuring disableBoiler is set in the querystring: /api/version/resource?disableBoiler. With that set, only the data object would be returned in the body of the response.

Callback using JSONP

We support JSONP, and to achieve that 3 things are needed:

  1. The ability to wrap the returned JSON in a callback function.
  2. The ability to only ever return a HTTP 200 OK status.
  3. The ability to perform other HTTP verbs via HTTP POST.

You can wrap the response in a callback function by adding callback=funcName to the querystring of your request.

You can overload the HTTP method using HTTP POST and then adding method=PUT or method=DELETE as necessary. Note: If you are not using JSONP please don't do this, just use the correct HTTP verbs.

By adding always200 to the querystring, the API will only ever return a HTTP 200 OK header, however the status property in the boiler plate JSON will still give the underlying HTTP status code.

So for GET and POST, you merely have made a request to a resource similar to /api/version/resource?always200&callback=funcName and for PUT and DELETE you have also added &method=PUT or &method=DELETE as necessary.

Note: You should not combine disableBoiler with the JSONP features, as the API will only ever be returning HTTP 200 OK, and you will need to check the status property of the boiler plate before handling the data.

You can also provide context via the querystring and whatever the value of that parameter is will be provided back to you in the context property of the boiler plate.
